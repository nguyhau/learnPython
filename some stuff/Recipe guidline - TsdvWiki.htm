What is recipe

Recipes are metadata files which contains information about Linux packages, mostly denoted by .bb extension (other extensions could be .bbappend and .inc). Information in recipe will then be used by Poky tool from Yocto project to build Linux packages. Recipe itself can be considered as a scripting language, however, their functions should be written in Shell or Python.

More detail explanations about recipe can be found here [1].

Recipe's structure

Basic format of recipe should contain following information: 

#
# Base recipe:
# Base branch:
# Base commit:
#
SUMMARY = “”
DESCRIPTION = “”
HOMEPAGE = “”
inherit debian-package [other stuffs]

PR = “”
DEPENDS = “”

LICENSE = “”
LIC_FILES_CHKSUM = “”

Base recipe information

In recipe, information about base recipe (recipe that being used as reference) should be described. To describe this information without affect recipe itself, use comment in recipe. 

#
# Base recipe: 
# Base branch:
# Base commit:
#

Note: Line in recipe which began with a sharp (#) is comments. However, sharp is one-line comment, Poky recipe doesn’t use multiple-lines comment. In these lines, following information should be included:

  • Base recipe: Path to base recipe with meta folder at beginning. This is a required line. 
Ex: Base recipe: meta/recipes-core/zlib/zlib_1.2.8.bb
  • Base branch: branch in git that base recipe located. This is a required line. 
Ex: Base branch: daisy
  • Base commit: Current commit at that branch, or commit at that base recipe was used. This is a recommend line. 
Ex: Base commit: 9e4aad97c3b4395edeb9dc44bfad1092cdf30a47

Package information

In recipe, each package has their own basis information or usage, include this information in recipe can help user to know their functions. 

SUMMARY = “”
DESCRIPTION = “”
HOMEPAGE = “”
…
DEPENDS = “”

LICENSE = “”
LIC_FILES_CHKSUM = “”

Note:

    • Words with all capital letters in recipe are considered recipe’s variables. These variables can be assigned value like string
    • All licenses required by packages should be listed in the recipe.

In these lines, following information should be included:

  • SUMMARY: General information about package’s functions. This is a recommend line. 
Ex: SUMMARY = "Theora Video Codec"
  • DESCRIPTION: Detail explanation about functions of packages. This is a recommend line. 
Ex: DESCRIPTION = “The libtheora reference implementation provides the standard encoder and decoder under a BSD license.”
  • HOMEPAGE: Home page of that package. This is a recommend line. 
Ex: HOMEPAGE = “http://xiph.org/”
  • LICENSE License of that package. This is a required line. 
Ex: LICENSE = "Zlib"
  • LIC_FILES_CHKSUM: contains md5sum of license file. md5sum of all license files should be listed here. This is a required line(s). 
Ex: LIC_FILES_CHKSUM = " \
        file://LICENSE;md5=44bc22578be94b6536c8bdc3a01e5db9 \
        file://png.h;endline=15;md5=aa42e53406a5ebb9500bae72fa314b6d \
        file://png.h;beginline=209;endline=323;md5=83e13159b5bf0a4d9fb634acbda7cb5b"

Inherit existed class

In Poky tool, some classes were developed for support build package procedure. For example, inherit autotools class will support build packages by autotools. In order to use these classes, use inherit keyword provided by Poky tool. 

inherit debian-package [other stuffs]

Note: inherit keyword specify which class will be used to build package. Classes in Poky normally located in meta/classes directory, with .bbclass extension. To inherit a class, just use file name without extension. For example, class autotools.bbclass can be inherit by: inherit autotools.

In that line, following information should be included:

  • inherit: class which support build package. inherit debian-package is required.debian-package class is required, other classes might be inherit as well if needed. 
Ex: inherit debian-package autotools.

Package Revision

In recipe, to maintain revision of recipe or package, PR variable is used. 

PR = ""

Note: This variable contains revision of recipe, set it to “r0” when initialize recipe and increase to “r1”, “r2” when modify recipes. In that line, following information should be included:

  • PR: package revision or recipe revision – this variable is used to keep track of changes in recipes during development phase. This line is required. 
Ex: PR = "r0"

Note: Information about recipe variables and keywords can be found in here [2]

Development Policies

Syntax

Recipes developed by SKERLET-SWC team should follow these syntax policies:

  • Recipe name
    • Recipe name should be in this format:
      • <Package name>_debian.bb
      • <Package name>: Contain name of package only, not included version number. 
Ex: zlib_debian.bb
  • Indentation
    • Shell function should use tab (width = 8) as an indentation
    • Python function should use 4 spaces not tab with width 4 as an indentation. 
Ex: Shell function
do_install_append() {
	install -m 0755 ${WORKDIR}/app ${D}${bindir}
}

Ex: Python function
python do_patch_append() {
   import subprocess
   subprocess.call("ln -sf ${STAGING_INCDIR}/linux ${S}/include/linux", shell=True)
}
  • Comment
    • Comments should be included in:
      • New variables, function in recipes (in compare with base recipe).
      • Overwritten, exclude variables in recipes (in compare with base recipe).
      • Comments inside functions are recommended.
  • Commits
    • Commit's subject should follow bellow policies:
      • Subject of message should begin with changed file
      • Purpose of commits should be explained clearly
      • Commit with sign-off option in git (option -s)

Development

Developing recipe in SKERLET-SWC team should follow these policies:

  • Licenses
    • Do not use license from base recipe directly. Re-check all licenses information required by packages and add them to recipe.
  • Debian packages
    • Debian package file (.deb file) built by SKERLET-SWC team’s recipes should:
      • Have the same name as Debian package in Debian Jessie
      • Contain same essential files, directories in each .deb file
      • Install binaries, libraries… in same path as in Debian Jessie packages.
      • Remove unneeded files, directories which were not suitable for embedded environment. For example: documents, manual, examples, tests…
  • Debian configurations
    • Configuration used in recipes should follow these policies:
      • Use same configuration option as Debian package by default.
      • Add other options if needed to build by Poky tools. For example: sysroot related options.
  • Applying patches
    • Applying patches in recipes should follow these policies:
      • Don’t include local files that also existed in Debian.
      • Don’t include bug fix patches.
      • Don’t include security fix patches.
      • Only include patches that required to build using Poky.
  • Dependency
    • Unneeded dependencies should be removed.
  • Package revision
    • Package revision should be increased whenever recipes had been modified.
  • File .inc
    • Only create .inc file when it’s sufficient. Do not create wasted .inc file because .inc could not work without .bb file. For example: .inc file contains almost all information and there are only few lines in .bb file. In such case, create only .bb file.

FAQs

How to check if packages existed in Debian Jessie?

Debian packages can be check online in: Debian package management This webpage provide search options to search using source package name, package name or package content.

How to retrieve dependencies in Debian packages?

All dependencies of a Debian package were listed in control file. This file located in folder debian inside every Debian source packages.

How to check installation and configuration of Debian package?

Configuration and installation of Debian package were listed in rules file. This file located in folder debian inside every Debian source packages. Some Debian packages even contains file named postinst, postinst.in which will be used when installing .deb file. Usually, these files contain information about install path, soft links…

How to change Debian package name, i.e .deb file name?

Poky tool automatically set package name according to Debian naming policy when specify PACKAGE_CLASSES as package_deb. However, it’s not correct sometime. So to change Debian package name, DEBIANNAME variable could be used. In few cases, this variable required LEAD_SONAME variable has to be set in order to change package name. In case, both of above choices could not work, then package list should be re-defined by PACKAGES variable. Remember to re-assign files into each package to avoid error: “install but not shipped”.

How can I check latest recipes in Openembedded online?

Meta-debian developed by SKERLET-SWC team was created as a meta layer which based on meta from Openembedded. All recipes from Openembedded located in meta folder is managed by git, it means that command git pull can fetch latest source code from Openembedded. It’s also available online at: Meta gitweb

Is there any official document for Poky tool?

Poky tool was developed by Yocto project, its home page is at: Yocto Home page Meta-debian was developed using Poky version 1.6 – code name Daisy. All documents for this version are available at: Poky Daisy Documents

There are way too many documents, do I have to read them all?

Yes and No. In case you want to have a deep knowledge about how Poky work, you should read them all. Otherwise, only 4 documents are recommended for developers:

  • BitBake User Manual: This document is good for developer who starting to develop recipe. This document will provide basis information of bitbake tools.
  • Development Manual: This document provides information of development process for images and user-space applications.
  • Linux Kernel Development Manual: This document provides information of Linux kernel development.
  • Reference Manual: This document provides a complete reference to Yocto Project tools. All variables and keyword can be found here.

How can I create patches file efficiently?

The most efficient way to create patch file is inside build tree folder after do_fetch, do_apply_patch have finished. Using this way, newly created patch will be on top of all applied-before patches.

Is there anyway to modify configurations other than options provided by packages?

All packages have limited options to modify its output. Beside these options, a directly modify into package’s configure files usually a good way. Package configuration files are differ among packages, however, a lot of them was build using autotools so Makefile, Makefile.am, configure.ac, configure,… are good place to modify. Developer might create patches to apply changes or modify directly inside recipe be creating function and use some Linux command like sed, grep, awk, cat… to modify.
